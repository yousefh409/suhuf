# Testing Strategy

Generated by `/pipeline-test-setup` — edit as needed.

## Project Profile

Three components under one repo:

| Component | Platform | Framework | Status |
|-----------|----------|-----------|--------|
| Recitation Engine | Python API | FastAPI + PyTorch | Active |
| Reader App | Mobile (iPad) | Expo / TypeScript | Planned |
| Website | Web | TBD (Next.js likely) | Planned |

- **CI:** GitHub Actions
- **Env vars:** None required — tests are self-contained
- **Testing philosophy:** Prefer AI-powered testing (MCP headless browser, `/qa`, `/browse`) over traditional scripted tests. AI analysis catches more nuanced issues and adapts to UI changes without brittle selectors.

## Test Types

### Unit Tests

**Recitation Engine (Python)**
- **Command:** `cd recitation && python -m pytest tests/ -v`
- **Framework:** pytest (to be added — currently script-based)
- **Covers:**
  - `arabic.py` — diacritic stripping, i3rab/tashkeel alternative generation, sukoon variants
  - `engine.py` — CTC scoring math, forced alignment, greedy decode, audio loading
  - `scorer.py` — MixGoP feature extraction, GMM scoring
  - `server.py` — `classify_words()` rules, threshold logic, word assessment
- **Migration path:** Extract testable functions from `evaluate.py`, `test_mutations.py` into pytest

**Reader App (Expo/TypeScript)**
- **Command:** `cd reader && npx jest --passWithNoTests`
- **Framework:** Jest (Expo default)
- **Covers:**
  - Audio recording hooks, playback state
  - Arabic text rendering logic
  - WebSocket client message parsing
  - Navigation state, passage selection
  - RevenueCat subscription state

**Website**
- **Command:** `cd website && npx vitest run`
- **Framework:** Vitest (or Jest depending on framework choice)
- **Covers:**
  - Page components, forms, CTAs
  - Pricing logic, feature gates
  - SEO metadata generation

### Integration Tests

**Recitation Engine**
- **Command:** `cd recitation && python -m pytest tests/integration/ -v`
- **Setup required:**
  - Models present in `recitation/models/` (not downloaded in CI — skip model-dependent tests)
  - ffmpeg installed
- **Covers:**
  - `POST /api/score` — upload audio → get scored words
  - `GET /api/passages` — passage loading from JSON
  - `POST /api/save` — recording save + manifest append
  - `GET /api/recordings` — manifest reading
  - Audio pipeline: webm → ffmpeg → PCM → scoring

**Reader App**
- **Command:** `cd reader && npx jest --config jest.integration.config.js`
- **Setup required:**
  - Recitation server running (or mocked)
  - Supabase local dev (or mocked)
- **Covers:**
  - WebSocket streaming session lifecycle
  - Audio capture → server → scored response
  - Supabase auth flow
  - Offline → online sync

**Website**
- **Command:** `cd website && npx vitest run --config vitest.integration.config.ts`
- **Covers:**
  - API routes (if any server-side)
  - Form submissions, email capture
  - Analytics event firing

### E2E Tests — AI-Powered (MCP Browser)

All E2E testing uses the MCP headless browser via gstack skills. AI-driven tests provide better analysis than scripted selectors — they understand visual context, can catch layout regressions, and adapt to UI changes.

**Recitation UI**
- **Tool:** `/qa` or `/browse` (MCP headless Chromium)
- **Target:** `http://localhost:8000`
- **Flows:**
  - Select passage → record audio → see scored results
  - WebSocket streaming session (record → live feedback → stop)
  - Error states: no microphone, invalid passage, network disconnect
  - Arabic text rendering and diacritic display correctness

**Website**
- **Tool:** `/qa` or `/browse` (MCP headless Chromium)
- **Target:** `http://localhost:3000`
- **Flows:**
  - Landing page → pricing → signup flow
  - Responsive layout checks (desktop 1440px, tablet 768px, mobile 390px)
  - CTA interactions, form validation
  - SEO: meta tags, OpenGraph, structured data

**Reader App (Expo)**
- **Tool:** `/qa` via Expo web preview + MCP browser
- **Target:** Expo web dev server (for visual/layout testing)
- **Flows:**
  - Onboarding → passage selection → recording → results
  - Subscription purchase flow
  - Offline mode indicators
  - Arabic text rendering with diacritics
- **Note:** Native-specific flows (audio permissions, background audio, haptics) require manual testing on iOS Simulator

## QA Phase Behavior

When `/pipeline-qa` runs:
1. Checks which components exist (have source files)
2. Skips components not yet built
3. Runs unit/integration tests via CLI commands
4. Runs AI-powered E2E via `/qa` on running dev servers
5. AI analyzes screenshots, DOM state, console errors, network requests
6. Reports by severity: critical, important, minor
7. Offers to fix critical/important issues
